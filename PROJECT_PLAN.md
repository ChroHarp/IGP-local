# IGP Local 專案計畫

> 更新日期：2026-07-15。本文以目前 repository、migration 與測試為準，作為 Phase 3 開始前的基準。

## 1. 產品目標與範圍

IGP Local 是供校內資優教育團隊使用的本機優先 IGP 管理系統。核心目標是讓特教組長、個管教師與任課教師在熟悉的 Django Admin 介面中，安全地管理學生資料、年度與學期計畫、課程計畫及學習成果。

目前核心範圍：

- 校內登入與角色權限。
- 學生、家庭、鑑定與 IGP 基礎資料。
- 年度、學期、課程計畫及批次／群組作業。
- 學習表現與任課教師評分。
- Excel 基礎資料匯入。
- PDF／DOCX 課程文件的私有保存與下載。
- Phase 3 將加入的輔導紀錄工作流與稽核紀錄。

非核心或延後範圍：

- 家長／學生一次性表單改為選配，須經校方公網與資安審查後才啟用。
- 不建立公開學生入口、REST API、行動 App 或即時通知系統。
- 不在目前階段導入 React、Docker、Celery、Redis 或 PostgreSQL。
- DOCX 輸出與完整報表仍屬後續階段，不是 Phase 3 的前置條件。

## 2. 技術與安全基準

- Python 3.13、Django 5.2、Django Templates、少量 JavaScript。
- SQLite 作為本機資料庫；正式環境由作業系統磁碟加密、備份與存取控制保護。
- `django-allauth` 僅用於核准帳號的 Google 登入，權限只申請基本身分與 Email。
- 真實學生資料、SQLite、附件、OAuth secret、log 與 backup 不得進入 Git。
- 所有學生資料查詢必須經集中式 policy 或可驗證的 queryset 篩選。
- 技術系統管理員不因技術角色自動取得學生資料權限；superuser 僅作離線 break-glass 帳號。
- 新增或修改資料模型時必須建立新 migration，不修改已套用的 migration。

## 4. Phase 3：輔導紀錄與稽核

Phase 3 的目的不是再增加一組一般 CRUD，而是建立可追蹤、可審核、不可任意回寫的校內紀錄流程。

### 3A. 規格與資料模型

先完成並以測試固定以下規格：

- `CounselingRecord` 至少包含學生、紀錄日期、紀錄者、主題／類別、內容、狀態及建立／更新時間。
- 狀態採 `draft -> submitted -> reviewed -> locked`，不得任意逆向跳轉。
- 明確定義每一角色在每一狀態可查看、建立、修改、送審、審核與鎖定的權限。
- 紀錄者只能修改自己的草稿；送審後內容不可直接覆寫。
- 被指派學生的可見性沿用現有有效指派與結束日期規則。
- 決定退回修正的方式：採獨立 `returned` 狀態，或保留送審狀態並建立新版；實作前必須先定案。
- 決定附件是否納入 Phase 3。若納入，只重用私有檔案驗證與受控下載，不提供公開 URL。

3A 驗收條件：

- 權限矩陣及狀態轉換表寫入文件或程式註解。
- model constraints、validation 與 migration 完成。
- 每種角色至少有正向與拒絕案例測試。

### 3B. Admin 工作流程

- 依目前登入者及學生可見範圍顯示輔導紀錄。
- 草稿編輯、送審、審核與鎖定使用明確動作，避免只靠一般儲存按鈕改狀態。
- 已送審或鎖定紀錄使用唯讀欄位；不得透過直接 Admin URL 繞過限制。
- 列表可依學年度、學生、紀錄者、狀態與日期篩選。
- 學生頁面可查看該生紀錄摘要，但仍套用相同權限。
- 錯誤訊息應指出是權限、狀態或資料驗證問題。

3B 驗收條件：

- 完整走通「建立草稿 → 送審 → 審核 → 鎖定」。
- 未授權使用者在列表、詳細頁及直接 URL 都不能讀取紀錄。
- 鎖定後不能修改或刪除；管理者若需更正，必須另建更正紀錄或使用明確且可稽核的流程。

### 3C. AuditEvent 最小可用範圍

`AuditEvent` 第一版只記錄高價值事件，避免一開始做成難以維護的全站資料鏡像：

- 輔導紀錄建立、送審、審核、退回（若採用）與鎖定。
- 帳號角色、核准狀態及學生教師指派變更。
- Excel 匯入確認寫入。
- 私有文件上傳與刪除。

每筆事件至少保存發生時間、操作者、事件類型、目標 model／主鍵及必要摘要。不得在 audit payload 複製完整敏感內容；AuditEvent 預設不可由一般 Admin 修改或刪除。

3C 驗收條件：

- 關鍵事件在同一 transaction 成功時才建立 audit；主操作失敗時不得留下誤導事件。
- 稽核列表只開放特教組長與經明確授權者。
- 測試確認事件內容不包含密碼、token、完整輔導內文或上傳檔案內容。

## 6. 實作原則與變更管理

- 每個階段先寫權限與資料生命週期，再寫頁面。
- 權限規則集中在 policy、model constraints 及 Admin queryset，不在模板中視覺隱藏代替授權。
- 所有新功能使用合成資料測試；不得將真實學生資料放入 fixture、測試或截圖。
- 每個 migration 必須可在現有資料庫副本演練，資料 migration 必須有明確的正向及必要時反向策略。
- pull 新版本不代表完成更新；資料庫 migration、相依套件、重啟與 smoke test 都是同一個 release 步驟。
- 一次提交只處理一個可說明的功能或修正，UI 變更附截圖，資料模型變更附 migration 與驗證命令。
- Phase 3 中若發現 Phase 2 缺陷，只修復阻斷性、資料完整性或權限問題；其他改善排入 backlog，避免範圍擴張。
